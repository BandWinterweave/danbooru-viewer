# 阶段 5 产品决策记录

本文记录阶段 5.4 中三项评估任务的当前结论。结论基于现有产品范围、实现成本和可验证性，不代表永久排除；重新评估必须满足各节列出的触发条件。

## Danbooru Notes

**决策：暂不实施。**

当前查看器统一支持五个图源，而 Notes 只由 Danbooru 提供。若形成完整 UI，需要额外处理图片原始坐标到预览、样图和缩放视口的坐标映射，并保证平移、缩放及响应式布局下 overlay 始终对齐。Note 正文可能包含 HTML，因此还必须定义严格的净化规则和 Content Security Policy 边界，防止 XSS；读取与编辑权限、认证失败语义、分页及本地缓存失效也需要单独设计。这些复杂度目前缺少真实使用需求支撑。

仅在收到明确的只读 overlay 需求和真实用户反馈后重新评估。届时先限定 Danbooru、只读展示、坐标映射和 HTML 净化方案，并建立安全及缩放测试。Notes 编辑涉及权限、冲突处理和写入审计，应作为独立项目规划，不与只读展示捆绑。

## 幻灯片

**决策：维持阶段 1 的下线状态，不恢复旧入口。**

旧入口没有完整可用的播放流程，保留或恢复只会再次产生失效功能。只有出现明确用户需求，并完成播放范围与顺序、循环和暂停规则、键盘交互、详情/侧栏冲突处理、媒体预取及缓存预算设计后，才重新规划新的幻灯片能力。重新规划应从行为规范和验收测试开始，不复用旧入口作为占位。

## 下载队列与进度

**决策：本阶段不引入持久队列、跨重启恢复或字节级进度。**

当前批量下载由 `downloadPosts` 串行逐项调用 `downloadPost`，任一项抛错后整批立即终止；界面提示开始下载，但没有逐项结果、失败继续或重试失败项能力。这是当前实现事实，不将其描述为完整下载队列。

重新评估的触发条件是出现可重复的批量失败用户反馈。后续应先实现会话级任务状态：逐项成功/失败结果、单项失败后继续，以及仅重试失败项。持久队列和跨重启恢复只有在会话级模型稳定且存在明确需求时再评估。字节级进度必须接入并验证 `chrome.downloads.onChanged`（以及对应跨浏览器行为）后才可称为完成，不能用“已提交下载”或任务计数冒充下载进度。
