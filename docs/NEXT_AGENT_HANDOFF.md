# 下一 Agent 执行与交接流程

## 1. 使用方式

将本文档中的“启动指令”直接交给下一个 Agent。Agent 必须以 [`IMPROVEMENT_ROADMAP.md`](./IMPROVEMENT_ROADMAP.md) 为任务来源，每次只处理一个边界清晰、可以独立验证的任务组。

除非用户明确要求，不要一次实施整个路线图，也不要跳过验证后直接勾选任务。

## 2. 启动指令

```text
你将接手 Danbooru Viewer 项目的改进工作。

首先阅读：
1. docs/IMPROVEMENT_ROADMAP.md
2. docs/NEXT_AGENT_HANDOFF.md
3. README.md
4. package.json

随后检查 git status 和最近提交，识别已有未提交改动。不得撤销、覆盖或整理不属于本任务的改动。

本轮只选择路线图中当前阶段最靠前、尚未完成且可以独立验收的一个任务组。开始编码前：
1. 阅读相关实现和测试。
2. 验证路线图中的问题仍然存在。
3. 写出本轮范围、涉及文件、风险和验证命令。
4. 若产品决策不明确或会改变已有用户数据，先询问用户。

实施时遵循现有架构和代码风格，采用最小正确变更。同步补充与风险匹配的测试，不进行无关重构。

完成后必须：
1. 运行相关测试和类型检查。
2. 根据影响范围运行 Chromium 或 Firefox 构建。
3. 检查 git diff 和 git diff --check。
4. 只有验收标准真实满足时，才更新路线图复选框。
5. 输出变更摘要、验证结果、剩余风险和建议的下一任务。

不要自动提交、推送或发布，除非用户明确要求。
```

## 3. 固定执行流程

### 步骤 1：恢复上下文

- 阅读路线图中的当前阶段、前置依赖和验收标准。
- 阅读最近提交，确认项目当前进展。
- 执行 `git status --short`，区分已有改动与本轮改动。
- 不假设路线图描述仍然准确，先在源码中验证问题是否存在。
- 检查相关测试是否已经覆盖或与当前实现发生漂移。

**输出：** 用一段简短说明陈述当前状态和准备处理的任务。

### 步骤 2：选择本轮任务

任务选择遵循以下顺序：

1. 当前阶段尚未完成的 P0 任务。
2. 阻塞其他任务或自动化验证的 P1 任务。
3. 当前阶段剩余的 P1/P2 任务。
4. 只有当前阶段验收通过后，才进入下一阶段。

一个任务组应满足：

- 修改范围通常集中在一个行为或一条业务链路。
- 可以在本轮内完成实现、测试和验证。
- 失败时容易回退或继续修复。
- 不依赖尚未确认的产品决策。

不要把“阶段 1 全部完成”作为单轮任务。推荐粒度示例：

- 新安装默认仅启用 `General`，并保留已有用户设置。
- 修复客户端日期过滤导致的分页提前结束。
- 下线幻灯片入口、命令、文案和过期测试。
- 为后台 API 请求增加超时和 payload 校验。

### 步骤 3：建立任务基线

- 定位相关组件、store、service、Manifest 和测试文件。
- 记录当前行为、预期行为及最小复现方式。
- 先运行最接近该行为的已有测试。
- 若没有测试，先定义可观察的验收条件。
- 检查该变更是否影响持久化数据、权限、隐私或跨浏览器行为。

**停止条件：** 若发现路线图要求与当前实现冲突，或需要迁移已有用户数据但缺少明确策略，应先向用户说明取舍，不要自行猜测。

### 步骤 4：制定本轮计划

计划应明确列出：

- 本轮目标和不包含的范围。
- 预计修改的文件和模块边界。
- 实现顺序。
- 测试与构建命令。
- 可能影响已有行为的风险。

涉及三个及以上独立步骤时，使用任务清单并实时更新状态。

### 步骤 5：实施最小正确变更

- 优先复用现有类型、工具函数和状态模式。
- 不为假设中的兼容场景增加代码。
- 不顺手整理无关文件、依赖或格式。
- 不修改或回退其他人产生的无关工作区改动。
- 对网络、持久化和权限操作提供明确错误处理。
- 对首次安装与已有用户设置分别处理，避免覆盖持久化选择。
- 用户可见行为变化时同步检查 README、商店文案和 locale。

### 步骤 6：分层验证

按照从快到慢的顺序验证：

1. 相关单元测试或组件测试。
2. `npm run typecheck`。
3. `npm test`。
4. 相关 E2E 或浏览器冒烟测试。
5. `npm run build`。
6. 涉及 Manifest、权限或浏览器差异时运行 `npm run build:firefox`。
7. `git diff --check`。

若某项验证因环境限制无法运行，必须在交接报告中说明原因、影响和人工验证方法。不能把“未运行”描述为“通过”。

### 步骤 7：审查变更

- 阅读完整 `git diff`，确认不存在凭据、生成文件和无关格式化。
- 检查错误路径、空状态、竞态和已有用户迁移场景。
- 核对 Chromium 与 Firefox 的差异影响。
- 检查新增测试是否验证行为，而不是只验证实现细节。
- 确认文档、设置、快捷键和实际功能不存在新的矛盾。

### 步骤 8：更新路线图

只有满足以下条件才能将任务从 `- [ ]` 改为 `- [x]`：

- 实现已完成。
- 相关测试通过。
- 阶段验收条件中对应行为已验证。
- 必要文档已同步。
- 没有已知阻断问题被隐藏。

若只完成任务的一部分：

- 不勾选原任务。
- 在原任务下增加简短说明，记录已完成部分和剩余工作。
- 必要时将任务拆成更小的复选框，但不要改变原始目标。

### 步骤 9：交接报告

每轮结束时使用以下格式：

```markdown
## 本轮结果

- 完成：列出已经满足验收标准的任务。
- 修改：列出关键文件和行为变化。
- 验证：列出实际运行的命令及结果。
- 未验证：列出无法运行的检查及原因，没有则写“无”。
- 风险：列出仍存在的边界条件或兼容性风险。
- 工作区：说明是否存在本轮之外的未提交改动。
- 下一步：推荐一个边界清晰的后续任务，不直接开始实施。
```

## 4. 各阶段指挥重点

### 阶段 0：工程基线

- 先保证验证命令可重复运行，再清理测试。
- 不因测试过期而简单删除关键行为覆盖，应使用当前业务预期替换。
- 构建产物和截图不得意外进入提交范围。

### 阶段 1：发布阻断问题

- 优先顺序：安全默认值、分页正确性、请求校验、导入安全、失效功能下线、异步错误。
- 持久化设置变更必须区分首次安装和已有用户。
- 下线幻灯片时应同时检查 Manifest、后台、设置、locale、README、商店文案和测试。

### 阶段 2：质量门

- 依赖应分批升级，避免将大量破坏性变化合并为一次修改。
- CI 必须使用干净安装并验证两种浏览器构建。
- E2E 优先使用可控 API mock，真实网络检查不能作为稳定性阻断条件。

### 阶段 3：架构与性能

- 搜索会话、详情资源和缓存体系应分为不同任务组。
- 重构前先补竞态和故障测试。
- 性能改进需要可观测指标，不以“代码更整洁”作为完成标准。

### 阶段 4：交互与可访问性

- 使用桌面和移动端视口验证，不只检查静态代码。
- 键盘、焦点、Escape 和外部点击行为需要实际浏览器测试。
- 视觉变更应检查文本溢出、遮挡和响应式布局。

### 阶段 5：产品能力与本地化

- 收藏库、凭据测试和本地化分别实施，不混成一次大型改版。
- 本地化前先稳定文案键和主要交互。
- 幻灯片仍属于暂不实施范围，除非用户重新确认产品决策。

### 阶段 6：发布收口

- 发布前重新核对实际权限、网络请求和隐私说明。
- 不仅验证构建成功，还要验证产物可以被真实浏览器加载。
- 发布、提交、推送和商店上传均需用户明确授权。

## 5. Agent 决策边界

以下情况可以自主决定：

- 与现有风格一致的小范围实现细节。
- 为已确认行为补充测试和错误处理。
- 修正由本轮变更直接导致的类型或构建错误。
- 更新与本轮行为直接相关的文档描述。

以下情况必须先询问用户：

- 改变内容评级、安全或隐私默认值，已确认的“首次仅 General”除外。
- 删除或迁移已有用户的收藏、设置和缓存数据。
- 增加浏览器权限、host permission 或外部服务。
- 改变支持的图源、认证方式或远程写入行为。
- 重新引入幻灯片或扩大当前阶段范围。
- 采用会影响大量文件的新框架、状态库或格式化工具。
- 提交、推送、发布或上传扩展商店。

## 6. 首轮推荐指令

下一个 Agent 的首轮任务建议从安全默认值开始：

```text
按照 docs/NEXT_AGENT_HANDOFF.md 执行。

本轮只完成 docs/IMPROVEMENT_ROADMAP.md 中“阶段 1 / 1.1 内容安全默认值”：
- 新安装首次仅启用 General；
- 不覆盖已有用户持久化的评级选择；
- 增加首次安装、已有设置恢复和手动修改评级的测试；
- 同步相关 README 或产品说明。

先验证当前持久化和 hydration 行为，再实施最小变更。完成后运行相关测试、完整单元测试、类型检查和适用构建。只有全部验收通过后才勾选路线图任务。不要处理其他阶段任务，也不要提交或推送。
```
