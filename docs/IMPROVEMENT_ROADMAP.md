# Danbooru Viewer 项目改进路线图

## 1. 文档目的

本文档基于当前 `0.1.0` 版本的代码、测试、构建配置与产品能力，归纳项目下一阶段的改进方向，并将工作拆分为可独立验收的多阶段任务。

路线图优先保证内容安全、数据正确性和交付可靠性，再处理架构、性能与产品扩展。各阶段不绑定固定日期，可根据参与人数和发布节奏进一步拆分为迭代。

## 2. 当前项目概况

### 2.1 技术栈

- React 18、TypeScript 5.7、Vite 5
- Manifest V3，支持 Chromium 和 Firefox 构建
- Zustand 状态管理及浏览器扩展存储
- IndexedDB / `idb-keyval` 持久化收藏、缓存和下载历史
- TanStack Virtual 提供帖子列表虚拟化
- Vitest 与 Testing Library 单元测试
- Python Playwright 验收脚本，但尚未形成统一、可执行的 E2E 工程

### 2.2 已实现能力

- 统一浏览 Danbooru、Gelbooru、Safebooru、Yande.re 和 Rule34
- 标签搜索、自动补全、排除标签、快捷标签和筛选预设
- 评级、评分、日期、尺寸和排序筛选
- 网格、瀑布流、列表布局与无限分页
- 帖子详情、标签分类、评论、图集和父子帖子
- 本地及部分远程收藏、收藏分组和 JSON 导入导出
- 单张、批量、多规格媒体下载
- 图片渐进加载、缩放、拖拽和本地媒体缓存
- 亮色、暗色和跟随系统主题
- Popup、设置页及 Danbooru 页面增强脚本

### 2.3 当前基线

- 现有单元测试：9 个测试文件、33 个用例
- TypeScript 严格模式检查可通过
- 已配置 Chromium 与 Firefox 构建脚本
- 尚无 CI/CD、覆盖率门槛、统一 E2E 入口和自动发布检查
- 依赖审计存在严重、高危和中危问题，需要在升级后重新确认

## 3. 改进原则

1. **安全默认值优先：** 首次安装仅展示 `General` 内容，其他评级必须由用户主动开启。
2. **正确性优先于新增功能：** 先解决分页、竞态、错误处理和持久化问题。
3. **功能入口必须闭环：** 无可用实现的设置、快捷键和宣传内容应暂时下线。
4. **变更必须可验证：** 每项核心行为至少具备自动化测试或明确的浏览器验收步骤。
5. **缓存和异步任务必须有界：** 网络请求可超时、可取消，内存和磁盘缓存具有容量限制。
6. **保持跨浏览器一致：** Chromium 与 Firefox 构建均应进入持续集成和发布验收。

## 4. 优先级定义

| 优先级 | 含义 | 处理要求 |
|---|---|---|
| P0 | 内容安全、数据损坏或发布阻断问题 | 下一版本发布前完成 |
| P1 | 核心流程错误、明显体验断点或高风险工程问题 | P0 完成后优先处理 |
| P2 | 可维护性、可访问性和完整性改进 | 纳入中期迭代 |
| P3 | 产品增强和长期演进能力 | 在质量基线稳定后推进 |

## 5. 分阶段实施路线

### 阶段 0：建立可重复的工程基线

**目标：** 明确当前可用状态，建立后续改动的统一验证入口。

**任务：**

- [ ] 记录 Node.js、npm、浏览器和扩展测试版本。
- [ ] 在全新依赖安装环境执行类型检查与全部单元测试。
- [ ] 分别执行 Chromium 和 Firefox 正式构建并记录产物结构。
- [ ] 盘点现有 E2E 脚本，删除或改写互相矛盾的验收条件。
- [ ] 固化首次加载、搜索、分页、详情、收藏和下载的冒烟用例。
- [ ] 建立缺陷清单，将问题按 P0-P3 分类并关联到本路线图阶段。
- [ ] 将 `.tsbuildinfo`、验收截图等生成文件从版本控制噪声中清理出去。

**前置依赖：** 无。

**验收标准：**

- 在干净环境中可以按照文档完成安装、类型检查、单元测试和两种浏览器构建。
- 现有测试不存在相互冲突或引用已删除组件的断言。
- 基线命令失败时能够明确定位到具体步骤。

### 阶段 1：修复发布阻断问题

**目标：** 消除内容安全、分页正确性、输入校验和失效功能风险。

#### 1.1 内容安全默认值（P0）

- [x] 将首次安装的评级过滤默认值设为仅 `General`。
- [x] 只迁移未初始化的新用户，不覆盖已有用户保存的评级选择。
- [x] 为首次安装、旧设置恢复和手动开启其他评级增加测试。
- [x] 更新 README 和商店文案中的评级说明。

#### 1.2 日期过滤与分页（P0）

- [x] 修正 Gelbooru、Yande.re 等客户端日期过滤后的空页处理。
- [x] 当本地过滤结果为空但远端仍有数据时，继续请求后续页。
- [x] 设置最大连续空页或最大请求次数，避免异常 API 导致无限循环。
- [x] 区分“当前过滤页为空”和“远端数据已经结束”。
- [x] 增加空过滤页、连续空页、远端结束和请求失败测试。

#### 1.3 后台请求与消息校验（P0）

- [x] 为扩展后台 `fetch` 增加统一超时和取消机制。
- [x] 校验 `API_REQUEST` payload、HTTP 方法、headers 和 body 类型。
- [x] 捕获异步拒绝并返回统一的结构化错误响应。
- [x] 限制详情链接和媒体地址只使用允许的 `http` / `https` 协议。
- [x] 避免在请求去重键、缓存键和日志中保存完整认证信息。

#### 1.4 收藏导入安全（P0）

- [x] 定义并校验收藏导入文件 schema。
- [x] 限制导入文件大小、记录数量、URL 协议和字段长度。
- [x] 检测重复帖子、重复分组和无效关联。
- [x] 在完整校验和写入成功后再替换旧数据，失败时保留原数据。
- [x] 覆盖损坏文件、超大文件、非法 URL 和写入失败测试。

#### 1.5 下线未闭环的幻灯片功能（P1）

- [x] 移除设置页中的幻灯片间隔配置。
- [x] 移除 Manifest 中的 `toggle-slideshow` 命令。
- [x] 移除后台对应的消息发送逻辑和本地化文案。
- [x] 修正 README、商店说明和 E2E 中的幻灯片声明。
- [x] 清理确认无引用的旧灯箱依赖和代码。

#### 1.6 异步失败处理（P1）

- [x] 为快捷键下载、收藏、投票和初始化 hydrate 增加错误捕获。
- [x] 为剪贴板权限拒绝提供可见反馈。
- [x] 统一 API、IndexedDB、下载和权限错误的提示格式。
- [x] 增加全局 `unhandledrejection` 监测测试，确保关键流程计数为零。

**前置依赖：** 阶段 0 的基线用例。

**验收标准：**

- 新安装用户首次加载只看到 `General` 内容。
- 日期过滤不会因单个空页提前结束分页。
- 无效扩展消息不会导致 Service Worker 未处理异常。
- 收藏导入失败不会破坏已有数据。
- 产品中不再出现无法使用的幻灯片入口或宣传。

### 阶段 3：重构请求生命周期与缓存体系

**目标：** 解决搜索竞态、详情耦合和长期运行资源增长问题。

#### 3.1 搜索会话（P1）

- [x] 为每次搜索创建独立会话，固化图源、凭据、查询和分页状态。
- [x] 使用 `AbortController` 取消切源、改筛选和新搜索产生的旧请求。
- [x] 移除模块级共享分页 Promise，避免不同搜索会话相互影响。
- [x] 明确区分首次加载、后台刷新、追加分页和重试状态。
- [x] 增加快速切源、连续输入、重复分页和慢请求竞态测试。

#### 3.2 详情资源解耦（P1）

- [x] 将评论、关联标签、图集和父子帖子拆成独立资源状态。
- [x] 使用 `Promise.allSettled` 或等价策略避免单项失败清空全部结果。
- [x] 为每个资源提供独立加载、失败和重试状态。
- [x] 避免标签 enrichment 更新帖子对象后重复请求全部详情数据。
- [x] 将详情数据编排从大型组件中抽离为可测试的 hook 或服务。

#### 3.3 有界缓存（P1）

- [x] 将媒体缓存从条目数量限制升级为总字节预算与 TTL 双重限制。
- [x] 卡片优先缓存缩略图，按需缓存 sample 或原图。
- [x] 为 API 和标签建议缓存增加容量限制及定期清理。
- [x] 在淘汰、替换和页面卸载时调用 `URL.revokeObjectURL`。
- [x] 避免每次缓存写入都扫描全部 IndexedDB 数据。
- [x] 建立长时间滚动和大量唯一搜索的内存、磁盘占用测试。

#### 3.4 状态边界整理（P2）

- [x] 减少 store 之间直接读取和修改状态的交叉依赖。
- [x] 将图源能力、认证能力和查询上下文建模为明确接口。
- [x] 拆分过大的 `PostDetail` 和全局样式文件。
- [x] 只在能够降低重复和测试成本时引入新抽象。

**前置依赖：** 阶段 2 已提供竞态、故障和缓存测试基础。

**验收标准：**

- 快速切换图源或筛选不会显示旧请求结果。
- 一个详情接口失败不影响其他详情模块。
- 长时间浏览后 API 缓存、标签缓存、Blob URL 和 IndexedDB 占用保持有界。
- 搜索、详情和缓存逻辑可在不渲染完整页面的情况下单独测试。

### 阶段 4：完善核心交互与可访问性

**目标：** 修复移动端功能缺失、请求抖动和键盘操作障碍。

#### 4.2 筛选请求控制（P1）

- [x] 刷新时保留已有结果并展示非阻塞加载状态。
- [x] 避免输入多位数字时为每个中间值发起请求。

#### 4.3 对话框与卡片语义（P1）

- [x] 打开详情时将焦点移动到标题或关闭按钮。
- [x] 添加焦点陷阱，并使背景内容处于 `inert` 状态。
- [x] 关闭详情后将焦点恢复到触发卡片。
- [x] 移除卡片容器和内部按钮之间的嵌套交互冲突。

#### 4.4 键盘与弹层交互（P2）

- [x] 为所有弹层触发器补充展开状态和可访问名称。

#### 4.5 状态反馈（P2）

- [x] 计数使用实际可见帖子数量，而非未过滤总数。
- [x] 当所有帖子因预览不可用而隐藏时展示明确空状态。
- [x] 区分无搜索结果、仍在加载、网络失败和本地过滤为空。

**前置依赖：** 阶段 3 的搜索状态和资源状态已经明确。

**验收标准：**

- 移动端可以访问所有侧栏核心功能。
- 高级筛选不会因逐键输入产生连续请求。
- 搜索、补全、帖子详情和弹层可仅使用键盘完成。
- 对话框关闭后焦点位置可预测，背景内容不可被误操作。

## 6. 建议的阶段依赖

```text
阶段 0：工程基线
  -> 阶段 1：发布阻断问题
  -> 阶段 2：测试与交付质量门
  -> 阶段 3：请求、状态与缓存重构
  -> 阶段 4：交互与可访问性
  -> 阶段 5：产品能力与中文界面
  -> 阶段 6：发布与文档收口
```

阶段 1 与阶段 2 的部分工作可以并行，但在 E2E 预期尚未稳定前，不应大规模扩展产品功能。阶段 3 的重构应依赖阶段 2 的测试保护；阶段 5 的本地化应等待主要交互和文案稳定。

## 7. 质量指标

以下指标用于判断路线图是否产生实际效果，而不是只完成代码变更：

| 领域 | 建议指标 |
|---|---|
| 内容安全 | 新安装首次加载非 `General` 内容数量为 0 |
| 正确性 | 搜索、分页、切源和日期过滤关键场景全部自动化覆盖 |
| 稳定性 | 核心 E2E 中未处理 Promise rejection 数量为 0 |
| 网络 | 所有后台 API 请求具有超时、取消和结构化错误 |
| 性能 | 长时间滚动后缓存条目、总字节和 Blob URL 数量保持有界 |
| 可访问性 | 搜索、补全、详情和主要菜单可仅通过键盘完成 |
| 跨浏览器 | Chromium 与 Firefox 构建及核心冒烟测试均通过 |
| 供应链 | 不保留无风险接受记录的严重或高危依赖漏洞 |
| 文档 | README、商店说明和实际功能保持一致 |

## 8. 单项任务完成定义

每个复选框任务只有同时满足以下条件才能标记为完成：

- 实现代码已经合并且无无关重构。
- 新增或更新了与风险相匹配的自动化测试。
- 类型检查、单元测试和适用的 E2E 测试通过。
- Chromium 与 Firefox 的受影响流程均已验证。
- 用户可见行为、权限或隐私发生变化时，相关文档已经同步。
- 不遗留失效设置、快捷键、文案、依赖或测试断言。

## 9. 暂不纳入近期范围

- 暂不实现新的幻灯片功能；先完整下线现有失效入口，未来依据需求重新设计。
- 暂不引入遥测或行为分析；继续保持本地优先的隐私策略。
- 暂不为了形式统一而一次性重写全部状态管理或样式系统。
- 暂不新增更多 Booru 图源，优先稳定现有五个图源的行为。
- 暂不承诺固定阶段日期，待确认团队容量后再为阶段设置时间盒。

## 10. 推荐启动顺序

首个迭代建议只选择以下工作，控制变更范围并尽快形成安全基线：

1. 新安装默认仅启用 `General`。
2. 修复客户端日期过滤导致的分页提前终止。
3. 下线幻灯片设置、命令和宣传内容。
4. 修复后台请求超时与消息 payload 校验。
5. 统一最小可运行的 E2E 入口并接入 CI。

完成以上工作后，再进入搜索会话、详情资源和缓存体系重构。
